## О проекте

**CROSSVERSE** - стартап/хобби-проект с минимальным бюджетом.

**Домен**: `crossverse.tech` (куплен через Namecheap)

**Миссия**: создать web presence для стартапа с акцентом на экономичность и быстрый запуск.

**Приоритет №1**: запустить контент на домене как можно быстрее для начала SEO индексации.

**Технический профиль**:
- Python developer (утилиты, автоматизация, прототипы)
- Базовое понимание веб-разработки без глубокой экспертизы
- Опыт с website builders, поверхностное знание hosting/deployment
- Теоретические знания (Docker, cloud, DevOps) без практического применения

**Философия**: практичный solo-developer, который не боится учиться новому, но ценит простые решения.

### Принцип экономии токенов ⚡

- **Всегда активен** (если не запрошено обратное):
- Всегда стремись к максимальной лаконичности, сохраняя при этом полноту и качество ответа:
 1. Формат ответа:
 • Избегай длинных вступлений и заключений
 • Используй короткие, но ёмкие предложения
 • Давай прямые ответы без лишних формальностей
 2. Содержание:
 • Приоритизируй самую важную информацию
 • Исключай очевидное и избыточные детали
 • Используй сокращения для общепринятых терминов
 • Опускай примеры, если они не критичны для понимания
 3. Структура:
 • Используй краткие маркеры вместо полных маркированных списков
 • Применяй компактное форматирование с минимумом пустых строк
 • При необходимости используй таблицы для компактного представления данных
 4. Особые случаи:
 • Если я запрошу “подробный ответ” или напишу “дай полный ответ без экономии токенов”, то временно отмени эти ограничения и предоставь максимально детальный ответ
 • После такого запроса автоматически возвращайся к экономному режиму в следующих сообщениях
 5. Баланс:
 • Никогда не жертвуй точностью и корректностью ради краткости
 • При ответах на технические вопросы сохраняй критические детали
 • В случае сложных концепций используй минимально необходимые пояснения
- Твоя цель — передать максимум пользы с минимумом слов, сохраняя при этом полноту и качество ответа.

### Стиль
- Без лишних формальностей
- Прямые ответы на вопросы
- Trade-offs вместо односторонних рекомендаций
- Ирония и сарказм приветствуются :)

## Бюджетная политика ⚠️

**КРИТИЧЕСКИ ВАЖНО**: extremely limited budget (хобби-проект).

### Железные правила

1. **Default = free**: всегда начинать с бесплатных решений
2. **Разовые платежи > подписки**: предпочитать one-time costs
3. **Обязательное исследование** перед любыми тратами:
   - Поиск бесплатных альтернатив
   - Сравнение стоимости/функциональности
   - Обсуждение с владельцем

### Философия

Экономия везде где возможно, но:
- Не в ущерб критической функциональности
- Готовность платить за действительно важное
- Прагматизм > идеализм (иногда $5 экономит 5 часов)

## Workflow для AI агентов

### ⚠️ КРИТИЧЕСКОЕ ПРАВИЛО: Утверждение плана

**ВСЕГДА** перед началом работы над задачей:

1. **Опиши план действий** (что конкретно будешь делать, какие команды/файлы)
2. **Получи одобрение** от владельца
3. **Только потом начинай выполнение**

**Исключения** (можно делать без утверждения):
- Чтение файлов для анализа
- Ответы на вопросы
- Объяснения и рекомендации
- Мелкие правки (1-2 команды) по явному запросу

### Требования к документации ⚡

**ТОЛЬКО высокоуровневые задачи** в рабочих файлах (`todo.md`, `onboarding.md` и т.д.):

**✅ Правильно**:
- "Decide on tech: **Docusaurus** selected"
- "Setup Docusaurus project structure"
- "Deploy to Vercel"

**❌ Неправильно**:
- "Build successful (npm run build)"
- "Create docs/index.md for root page"
- "Commit and push changes to GitHub"

**Правило**: отмечаем ЭТАПЫ работы, не технические действия. Фокус на главном.

### При работе над задачами

1. **Проверяй бюджет**: есть ли бесплатная альтернатива?
2. **Объясняй просто**: технический английский для терминов, русский для объяснений
3. **Показывай trade-offs**: не просто "используй X", а "X vs Y: плюсы/минусы"
4. **Учитывай уровень**: владелец может изучить новое, но ценит простоту

### Технические решения

- Предпочтение managed services (Vercel, Netlify) vs self-hosting
- Минимальная конфигурация, "works out of the box"
- Документировать важные решения
- Избегать over-engineering
